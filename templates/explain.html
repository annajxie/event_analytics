About

My project studies PostgreSQL's cost-based Query Optimizer.

Background

PostgreSQL has one of the most transparent cost-based query optimizers of most open source databases. By using EXPLAIN and EXPLAIN ANALYZE queries, PostgreSQL allows us to see exactly what the planner decided, how much the estimated cost is, run time, etc. This transparency is the crux of my project, as it gives me a clear picture on how the cost-based query optimizer behaves.

The goal of the project is to understand how and why PostgreSQL makes its query execution choices based on its internal cost model. I have 8 benchmark queries that should in theory each activate a different query plan for PostgreSQL's optimizer so we can examine the cost breakdown and understand the motives for why and how the plans are chosen.

Set-Up

To work with I needed data for testing. I used Python's faker library to generate data for an event planning company. I attached the event dataset to a database schema I created in pgAdmin to interact with PostgreSQL. This website runs queries through connecting to that postgres database.

The events dataset consisted of two tables -- users and events -- where one user can attend many events but each event belongs to one and only one user. The user table consists of the name, email, region, as well as an id generated by pgAdmin. The events table consists of user_id, event_type, location, event_date, number of attendees, and status (active or inactive). I specified the specific ratio of event_type values in events table so that when filtering by group, I could more easily show how certain data criteria begets different query plan choices and why so many different plans exist and work.

Among other edits, I also adjusted random_page_cost from 4.0 to 2.0. RPC is the cost of reading one page of a PostgreSQL heap file through random disk access. The starting unit cost of 1.0 is a sequential read (reading a page beginning to end) and the reason why RPC is 4x the amount is because on a HDD, that was the expense conversion however on a SSD the two are nearly equal. I changed the value to make it easier to do index scans, where you search via a specific index of a value.

PostgreSQL is also capable of a feature known as MVCC (Multi-Version Concurrency Control) where instead of locking a row when someone updates it, where a new version of that row is created and the old version is still kept. To show MVCC, I needed two active mysql temrinals which was difficult to do on this website so it resides outside the current scope of my project right now.
