Sequential scan

Selecting rows where event_type = 'concert' returns too many rows for the index to help. PostgreSQL reads every heap page instead.

Seq Scan
SQL query
-- low selectivity → sequential scan EXPLAIN ANALYZE SELECT * FROM events WHERE event_type = 'concert';

Application mapping

Query: Retrieves all events of a common type to display on a dashboard.

Internal PostgreSQL: The optimizer estimates that ~50% of rows match. Fetching them via index requires thousands of random page reads — more expensive than reading all heap pages sequentially. It chooses Seq Scan.

Why it matters: Indexes are not universally beneficial — selectivity determines whether an index is worth using. This threshold is controlled by the random_page_cost parameter (default 4.0).
